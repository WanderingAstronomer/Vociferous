# Vociferous — Architecture & Scope

> **Status:** Authoritative. If a feature proposal contradicts this document, the proposal is wrong by default.

---

## 1. Mission Statement

**Vociferous turns an individual's speech into structured, refinable records.**

That sentence is the whole product. Everything in this codebase exists to serve it. Everything that does not is scope creep.

### 1.1 What "individual" means

Vociferous is a **single-user, local-first desktop application**. There is no concept of accounts, teams, organizations, tenants, or remote ownership. The user owns their data, the app runs on their machine, and the database lives on their disk. Sync, sharing, and collaboration features are explicitly **out of scope**.

### 1.2 What "speech" means

Vociferous captures **one human voice at a time**, in conversational, dictation, or monologue contexts. Multi-speaker audio, meeting transcription, speaker diarization, conference-room mic arrays, and team-call ingestion are **out of scope**.

If a user records a meeting, the result is a single undifferentiated transcript. That is by design.

### 1.3 What "structured, refinable records" means

A transcript is not the product. The **record** is the product:

- **Captured** — raw ASR output, immutable, with provenance (provider, model, runtime).
- **Titled** — short, scannable display name generated by an SLM.
- **Tagged** — user- and system-applied tags, including system tags like `Prompt`.
- **Refined** — variant text produced by an LLM, stored alongside the original (never overwriting).
- **Searchable** — full-text and tag-filtered.
- **Annotated** — insight summaries built across the corpus.

"Refinable" means the user can re-run refinement, edit, or replace variants. The original capture remains intact forever.

## 2. Hard Scope Boundary

### 2.1 In scope

| Concern | In scope because |
|---|---|
| Speech capture (hotkey, push-to-talk, chord) | Core mechanism for turning speech into a record |
| Local ASR (faster-whisper / CTranslate2) | Default capture path; no network dependency |
| Remote ASR (OpenAI-compatible, e.g. Groq) | User choice; same data flows into the same records |
| Local SLM (CTranslate2 Generator) | Default refinement and title generation |
| Remote refinement (OpenAI-compatible, LM Studio, Groq) | User choice; same variants stored in the same DB |
| Tagging, search, and editing | Structuring the record |
| Insight generation across the corpus | Adding structure across records |
| Persistence via SQLite (`TranscriptDB`) | The records have to live somewhere |
| Pywebview shell + Svelte 5 SPA frontend | The user has to see and control the records |

### 2.2 Out of scope

| Anti-feature | Why it stays out |
|---|---|
| Multi-speaker / diarization | Not what an individual's speech means |
| Meeting transcription with attendee tracking | Team workflow, not individual record-keeping |
| Real-time collaboration / shared corpora | Single-user product |
| Cloud sync, account-based backup | Local-first; user owns their disk |
| Agent-mode chat with the user's corpus | Belongs in a downstream consumer, not in Vociferous |
| Workflow automation / scheduling / cron triggers | Not what records do |
| Plugin marketplace, third-party extension API | Surface-area trap |
| Native mobile apps | Desktop dictation product |
| Web-hosted multi-tenant deployment | Single-user product |

### 2.3 The "is it an emitter or a router?" test

When a feature is proposed, ask: **"Does this turn speech into a structured record, or does it route records somewhere else?"**

- *Turns speech into a structured record* → consider it. May be in scope.
- *Routes records somewhere else, or consumes records to do something downstream* → out of scope. That belongs in a downstream consumer.

Vociferous is the **emitter** of speech-derived structured data. It is not a router, an agent, or a chat client. Downstream tools that consume the Vociferous corpus (e.g. Project Astronomer for personalized agentic responses) live in their own repositories, with their own runtime, and treat the Vociferous database as a read-mostly data source.

## 3. Architectural Invariants

These are **non-negotiable**. A change that violates one of these requires a documented architectural decision and a version bump.

### 3.1 Composition Root

`src/core/application_coordinator.py` is the **only** composition root. It owns:

- The `pywebview` window.
- The Litestar API server thread.
- The `CommandBus` and `EventBus`.
- All long-lived service instances (`TranscriptDB`, `RecordingSession`, `SLMRuntime`, `InsightManager`, etc.).

Services are constructed once at startup and injected. No service constructs another service in production code paths.

### 3.2 The H-Pattern (Intent-Driven Interaction)

All state changes must follow this path:

```
Frontend UI
  → POST /api/intents
    → CommandBus
      → Service logic (handler)
        → EventBus
          → WebSocket
            → Frontend store
```

**API handlers dispatch intents. They do not call services directly.** Services do not import from `src/api/`. The API layer is a transport adapter, not a business-logic layer.

### 3.3 Threading Model

| Thread | Owns | Forbidden |
|---|---|---|
| UI / main | `pywebview` window | Any blocking I/O, ASR, SLM, DB |
| API thread | Litestar event loop | Synchronous blocking calls in `async def` routes |
| Recording thread | Audio capture + ASR inference | UI calls |
| SLM thread | Refinement, title, insight inference | UI calls, concurrent SLM calls (lock-protected) |

Long-running work goes to a background thread immediately. The UI thread must never block.

### 3.4 Persistence

- All persistence flows through `TranscriptDB` (and its repository components).
- The **original raw transcript is immutable**. Edits and refinements are stored as variants.
- Audio assets are tracked by reference in the durable audio vault and linked to transcripts via foreign key.
- Migrations are append-only and live in `src/database/migrations.py`.

### 3.5 Provider Abstraction

ASR and refinement both support local and remote providers behind a single protocol per domain. Provider quirks (LM Studio reasoning routing, Groq parameter validation, MTP retry behavior) are isolated inside provider implementations, not leaked into handler logic.

The `application_coordinator` and handlers consume `RefinementProvider` / transcription provider interfaces. They do not know whether the call is going to a local CTranslate2 generator or a remote OpenAI-compatible endpoint.

### 3.6 Service Isolation Rules

- `src/services/` must not import from `src/api/`.
- `src/database/` must not import from `src/services/` or `src/api/`.
- `src/core/` is the integration layer; it can import from `services`, `database`, and `refinement`.
- `src/refinement/` is self-contained and imported by core; it does not import from core handlers.

## 5. Relationship to Downstream Consumers

### 5.1 Project Astronomer (and equivalents)

Project Astronomer is a **separate product** that uses the Vociferous corpus to personalize agentic responses. It treats the Vociferous SQLite database (or a read-only export) as input. It does **not** live in this repository.

The interaction model:

- **Vociferous emits.** It produces structured speech records.
- **Astronomer (or any downstream consumer) reads.** It queries the corpus, optionally subscribes to new-record events, and uses the data to inform its own behavior.

There is **no** API in Vociferous designed for agentic consumption. If a downstream consumer wants stable access, it gets it through:

1. Direct SQLite read access (the database schema is the contract).
2. An optional read-only export tool (future, not yet built).

Vociferous will **not** grow features to support downstream consumers. The data is the contract.

### 5.2 Why this matters architecturally

Treating downstream consumers as separate products keeps Vociferous small. The moment Vociferous starts shipping features "for Astronomer," the boundary collapses and the product becomes a hybrid that does neither job well.

## 6. When to Update This Document

Update `ARCHITECTURE.md` when:

- A new architectural invariant is added.
- An existing invariant is consciously relaxed (requires explicit version-bump documentation).
- Scope changes (rare; requires a deliberate decision, not a drift).
- A new module of significant size is added to the map.

Do **not** update this document to document tactical changes or refactoring within the existing architecture. Those belong in `CHANGELOG.md` and per-issue dossiers.
